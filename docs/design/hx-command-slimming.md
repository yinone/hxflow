# HX Command 瘦身方案

> 状态：proposal
> 日期：2026-04-01
> 目标：只优化 command contract，让 AI 更容易选对命令、抓住重点、减少误读

---

## 1. 问题判断

当前仓库的命令设计，主要问题不是命令解析机制，而是 command contract 本身信息密度失衡：

- 单个命令正文过长，重复了大量全局规则
- `输入 / 输出 / 约束` 中混入了太多共享 schema 细节
- 命令目标、适用场景、停止条件不够前置
- 阶段编号、共享 contract、固定字段说明在多个命令中重复出现
- AI 读完后更容易记住“规范细节”，而不是“这个命令到底该在什么场景下用”

结果是：

- 命令边界看起来很重，但不够清楚
- 相邻命令之间的职责差异不够显眼
- AI 在 `hx-go / hx-run / hx-ctx / hx-review / hx-clean / hx-qa` 之间更容易摇摆

---

## 2. 瘦身目标

本轮只做 command 层瘦身，不改以下内容：

- 不改 skill 安装模型
- 不改三层解析顺序
- 不改 protected / hooks 机制
- 不强行删命令

本轮只收敛 command contract 的表达方式：

1. 一个命令只表达一个用户意图
2. 共享规则上收，不在每个命令里重复展开
3. 先写“何时使用”，再写“怎么执行”
4. 只保留命令特有信息，不重复全局固定 schema
5. 输出重点从“字段清单”改成“产物和下一步”

---

## 3. 设计原则

### 3.1 先意图，后步骤

command 正文最先回答两个问题：

- 这个命令解决什么问题
- 什么情况下应该用它

如果 AI 先看到一长串依赖文件、schema 字段、固定输出格式，再看到命令目标，理解成本会明显上升。

### 3.2 共享事实不重复

以下内容应视为共享事实源，不再在每个命令里展开复述：

- `src/contracts/runtime-contract.md`
- `src/contracts/feature-contract.md`
- `src/contracts/progress-contract.md`
- `src/contracts/resolution-contract.md`
- 结构化结果固定字段顺序
- Hook / pipeline / protected 的共同行为

命令正文只写“需要用到哪份共享事实”，不重复“那份事实的完整规则”。

### 3.3 一个命令只保留自己的边界

命令正文里只保留三类边界：

- 触发边界：什么时候该用 / 不该用
- 数据边界：这个命令真正依赖哪些输入
- 行为边界：成功标准、失败停止点、下一步

### 3.4 约束只保留命令特有项

如果一个约束适用于大多数命令，就不应该重复写在单个命令里。

例如：

- 固定结构化结果字段
- 三层解析顺序
- 不做多层 merge

这些都应留在共享 contract。

### 3.5 命令说明面向决策，不面向存档

AI 读 command contract 的主要任务不是做制度学习，而是做执行决策。

所以命令文案应优先帮助 AI 做三件事：

- 选对命令
- 少走错分支
- 知道做完后该进入哪一步

---

## 4. 推荐模板

推荐把 command 正文收敛为以下结构：

```md
# <命令标题>

## 目标

- 这个命令解决什么问题

## 何时使用

- 适用场景
- 不适用场景

## 输入

- 只列命令特有参数
- 只列命令真正依赖的关键输入

## 核心步骤

1. 3 到 6 步，按决策顺序写
2. 共享规则只引用，不展开

## 成功结果

- 产物
- 对用户可见的结论

## 失败边界

- 直接阻断条件

## 下一步

- 默认推荐命令

## 约束

- 只保留命令特有约束
```

不再推荐在每个命令里重复：

- 固定结构化结果字段清单
- 所有默认路径展开
- 过多的“读取某文件头部某两个字段”式重复描述
- 与相邻命令重复的前置检查

---

## 5. 当前命令的瘦身方向

### 5.1 主链路命令

这类命令面向日常使用，正文要最短、最稳定：

- `hx-init`
- `hx-go`
- `hx-run`
- `hx-status`
- `hx-mr`

要求：

- 优先强调用户意图
- 减少阶段化叙述
- 不展开内部恢复、校验、复查机制的细枝末节

### 5.2 专家命令

这类命令保留，但应显式标记“用于拆开主链路或排障”：

- `hx-doc`
- `hx-plan`
- `hx-rules`
- `hx-fix`

要求：

- 正文中明确写“正常情况下优先使用 `hx-go` / `hx-run`”
- 只说明它们为什么独立存在

### 5.3 候选合并命令

这类命令最容易让 AI 混淆，应在后续阶段合并为统一检查意图：

- `hx-ctx`
- `hx-review`
- `hx-clean`
- `hx-qa`

建议方向：

- 短期：先重写 contract，明确差异
- 中期：合并为 `hx-check`

### 5.4 系统维护命令

这类命令不应与主链路命令放在同一认知层级：

- `hx-doctor`
- `hx-upgrade`
- `hx-uninstall`
- `hx-issue`

要求：

- 在帮助和文档中单独分组
- 正文避免混入日常研发语义

---

## 6. 推荐改写规则

对每个 command，建议执行以下改写：

1. 删除重复的固定输出 schema 描述
2. 删除与全局 contract 重复的解析规则
3. 把“何时使用”前移到正文前半部分
4. 把 `Rules Version / Updated At` 之类共享读取规则上收到共享 contract
5. 把默认下一步压缩为一句明确建议
6. 把执行步骤压缩到 3 到 6 步
7. 每个命令只保留 2 到 5 条特有约束

---

## 7. 实施顺序

### Phase A

先更新 command 编写规范：

- `src/contracts/command-contract.md`

目标：

- 先统一“以后怎么写”
- 避免继续生成臃肿 contract

### Phase B

按优先级改写主链路命令：

- `hx-go`
- `hx-run`
- `hx-doc`
- `hx-plan`
- `hx-mr`

目标：

- 先降低 AI 在主路径上的理解噪音

### Phase C

再处理辅助命令：

- `hx-ctx`
- `hx-review`
- `hx-clean`
- `hx-qa`
- `hx-fix`

目标：

- 收敛边界
- 为未来合并做准备

### Phase D

最后再看是否需要真正减少命令数：

- 若 AI 仍频繁混淆，再把 `ctx / review / clean / qa` 合并成 `hx-check`

---

## 8. 结论

command 瘦身的重点，不是先删文件，而是先让每个命令正文只表达：

- 目标
- 适用场景
- 核心步骤
- 成功结果
- 停止边界
- 下一步

先把 contract 写短、写清、写出边界，AI 的命令选择会先明显变稳。命令数量是否继续减少，可以放到下一阶段再做。
