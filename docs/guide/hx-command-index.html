Overview

命令总览

当前版本包含 3 个 CLI 子命令和 15 个 Agent 命令，数量以 bin/hx.js 和 src/commands/ 为准。主路径命令为 /hx-doc → /hx-plan → /hx-run → /hx-qa → /hx-mr。

命令 类型 分组 主要作用
hx setupCLI安装与维护手动重跑安装，修复 ~/.hx/、~/.claude/skills/ 与 ~/.agents/skills/
hx migrateCLI安装与维护将 1.x / 2.x 安装产物迁移到当前模型
hx versionCLI安装与维护输出 CLI 版本
/hx-initClaude项目初始化分析项目结构，写入 .hx/config.yaml，注入 CLAUDE.md 标记块
/hx-upgradeClaude安装与维护升级框架并同步安装产物
/hx-uninstallClaude安装与维护卸载安装痕迹
/hx-doctorClaude安装与维护检测环境、安装与项目配置健康状态
/hx-docClaude文档与计划获取需求并创建需求文档（团队模板）
/hx-planClaude文档与计划生成执行计划和 progress JSON
/hx-ctxClaude文档与计划当前需求执行前预检（可选诊断）
/hx-statusClaude文档与计划查看任务进度（进度条 + 状态汇总）
/hx-runClaude执行与复核默认执行整个需求；必要时可下钻到单个任务
/hx-goClaude执行与复核全自动流水线（读取 pipeline YAML 驱动，支持 --from 恢复）
/hx-reviewClaude执行与复核按团队审查清单审查 diff
/hx-fixClaude执行与复核读取 Review 意见按团队规范修复
/hx-qaClaude质量与交付运行团队对应的质量校验命令
/hx-cleanClaude质量与交付工程清理扫描（全局 + 团队专项）
/hx-mrClaude质量与交付输出 MR 创建上下文
Bootstrap

安装与维护命令

CLI 只保留安装入口和版本查询；其余维护动作改为 Agent 命令执行。

Setup

hx setup

hx setup [--dry-run]

手动重跑安装逻辑，修复固定的 skill 入口，初始化 ~/.hx/ 目录结构。

会创建 ~/.hx/（commands/、hooks/、pipelines/）目录结构。

同一套 workflow skill 会固定安装到两个入口：~/.claude/skills/ 与 ~/.agents/skills/。

~/.hx/settings.yaml 不存在时创建，只写入 frameworkRoot；后续修复直接按当前模型重跑。

安装后需要先手动运行一次 hx setup；后续则主要用于修复或补装。

Migrate

hx migrate

hx migrate [--dry-run]

将 1.x / 2.x 安装产物迁移到当前模型。

优先使用显式传入的目标入口。

未指定目标时，默认迁移 claude 与 agents 两个入口。

迁移时会重跑 hx setup，按当前模型重建入口并清理遗留安装字段。

Init（Claude）

/hx-init

/hx-init

在 Claude Code 或 Codex 中运行。分析项目结构，补齐项目配置、规则和目录骨架。

会写入 .hx/config.yaml（paths + gates）。

在 CLAUDE.md / AGENTS.md 中注入 harness 标记块。

生成 .hx/rules/*.md 项目规则文件骨架。

Version

hx version

hx version

输出当前 CLI 版本号。也支持 hx --version 与 hx -v。

Maintenance（Agent）

/hx-upgrade / /hx-uninstall / /hx-doctor

在 Claude / Codex 会话中运行

升级、卸载和诊断都通过 Agent 命令执行，不再依赖本地 CLI 子命令。

Planning

文档与计划命令（Agent 命令）

该组命令在 Claude Code 或 Codex 中运行，负责需求、计划与上下文准备。

Doc（Claude）

/hx-doc

/hx-doc

从当前输入或已连接需求来源生成结构化需求文档。

路径从 .hx/config.yaml 的 paths.requirementDoc 模板推断。

命令会基于需求详情整理需求文档内容，并生成项目内稳定的 feature 标识。

这个 feature 默认不需要你手动管理；后续命令会优先自动续接。

Plan（Claude）

/hx-plan

/hx-plan [<feature>]

读取需求文档，生成计划 Markdown 与 *-progress.json。

路径从 .hx/config.yaml 的 paths.planDoc 模板推断。

若省略 feature，会优先从当前目标需求文档头部的内部标识自动解析。

生成后会写入计划文档和 *-progress.json，后续命令直接以此作为状态来源。

Ctx（Claude）

/hx-ctx

/hx-ctx [<feature>]

面向当前需求做执行前预检，主要用于排查输入不完整或规则文件缺失。

正常主流程不用先单独运行；/hx-run 会自动完成同等预检。

仅在 /hx-run 因文档、计划或 progress 文件异常而停止时，用它查看具体缺失项。

Status（Claude）

/hx-status

/hx-status [<feature>]

查看任务进度。

无参数时展示所有需求的进度条汇总。

指定 feature 时列出所有任务的详细状态。

高亮下一个 pending 任务，并默认提示继续运行当前需求对应的命令。

Execution

执行与复核命令

该组命令在 Claude Code 或 Codex 中运行，负责执行任务与复核结果。

Run（Claude）

/hx-run

/hx-run [<feature>] [--plan-task <TASK-ID>]

默认执行当前需求下的全部 pending 任务；--plan-task 仅用于高级模式。

从 .hx/config.yaml 的 paths.* 字段读取需求文档、执行计划和进度文件路径。

省略 feature 时，会优先从当前需求/计划上下文自动续接。

默认按计划顺序逐任务推进，并在每个任务完成后自动更新 progress。

需要调试、重试或断点恢复时，再传 --plan-task 下钻到单个计划任务。

Go（Claude）

/hx-go

/hx-go [<feature>] [--from <step-id>]

全自动流水线，默认串起主路径；项目级 pipeline 可插入 review、clean 等辅助步骤。

按 src/pipelines/README.md 的公共规则读取 pipeline YAML（项目层 → 用户层 → 框架层），按步骤驱动执行。

--from <step-id> 支持从指定步骤恢复（如 --from run 跳过 doc/plan）。

run 步骤默认执行整个需求；需要下钻单任务时应直接使用 /hx-run --plan-task ...。

默认主路径为 doc -> plan -> run -> qa -> mr；更多步骤由 pipeline 覆盖控制。

默认主路径只保留文档与计划两个确认点，后续可由项目级 pipeline 自行追加。

Review（Claude）

/hx-review

/hx-review

按团队审查清单审查 diff。

读取 .hx/rules/*.md 项目规则，在基础清单后追加团队专属检查项。

Fix（Claude）

/hx-fix

/hx-fix

读取 Review 意见按团队规范修复。

会加载 .hx/rules/golden-rules.md 和 fix Hook。

Quality & Release

质量与交付命令

该组命令在 Claude Code 或 Codex 中运行，负责执行后的校验和交付。

QA（Claude）

/hx-qa

/hx-qa

从 .hx/config.yaml 的 gates 中按顺序执行质量校验。

固定顺序 lint → build → type → test，只执行已定义步骤。

任一步骤失败后停止，输出失败详情供 /hx-fix 使用。

Clean（Claude）

/hx-clean

/hx-clean

工程清理扫描（全局 + 团队专项）。

扫描 src/ 下的 console.log、: any、裸 throw new Error、TODO 等模式。

.hx/rules/ 中的项目规则可追加专项扫描约定。

MR（Claude）

/hx-mr

/hx-mr [<feature>] [--project <group/repo>] [--target <branch>]

输出 MR 创建上下文。

汇总需求文档、任务完成情况、git log 和 diff stat。

省略 feature 时，会优先从最近一次执行上下文自动续接。

默认目标分支是 main；--project 用于显式指定目标仓库。
