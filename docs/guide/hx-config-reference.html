Precedence

配置入口与优先级

用户级 settings 和项目级 config 是两套独立入口，不做字段合并。

配置文件

两个正式入口

~/.hx/settings.yaml
<project>/.hx/config.yaml

前者承载用户级安装设置，后者承载项目级运行时配置。

优先级

资源解析顺序

项目级 commands/hooks/pipelines
→ 用户级 commands/hooks/pipelines
→ 框架内置资源

所有命令先继承 src/contracts/runtime-contract.md；实体解析顺序统一看 src/contracts/resolution-contract.md；hooks 不覆盖，而是按固定顺序组成中间件链；~/.hx/settings.yaml 不参与项目配置合并。

i
边界：~/.hx/settings.yaml 只记录安装级设置；项目运行时事实由 .hx/config.yaml 维护。
Schema

.hx/config.yaml 核心字段

项目级 .hx/config.yaml 当前只维护 paths 和 gates 两组字段。

object

gates

定义质量门命令，按 lint → build → type → test 顺序执行。留空表示当前项目未启用该检查项。

gates:
  lint: npm run lint
  type: npm run type-check
  test: npm run test
object

paths

所有字段均为路径模板，支持 {feature} 占位符。其中 {feature} 表示当前需求在项目内的稳定标识。初始化后会显式写出默认值，便于后续编辑。

paths:
  src: app                           # 源码目录（默认 src）
  requirementDoc: docs/req/{feature}.md   # 需求文档模板
  planDoc: docs/plans/{feature}.md        # 执行计划模板
  progressFile: docs/plans/{feature}-progress.json  # 进度文件模板
Paths

路径模板字段

所有 paths 字段均为路径模板，支持 {feature} 占位符替换。其中 {feature} 表示当前需求在项目内的稳定标识。下面出现的 docs/requirement 与 docs/plans 仅表示默认值，不是强制目录。

默认值

paths 的默认值（未写入时生效）

requirementDoc → docs/requirement/{feature}.md
planDoc        → docs/plans/{feature}.md
progressFile   → docs/plans/{feature}-progress.json
src            → src
覆盖方式

推荐显式写出最终值

paths:
  requirementDoc: 需求/{feature}.md
  planDoc: 计划/{feature}.md

当前实现会在 .hx/config.yaml 中显式写出最终值；命令运行时直接按这些路径模板定位需求、计划和进度文件。

!
约定：/hx-init 会自动检测项目目录结构，并把最终生效的 paths 配置显式写入 .hx/config.yaml。
Hooks

Hook 规则

Hook 不再写在 .hx/config.yaml 中，而是由命令 frontmatter 和固定文件命名共同决定。

启用条件

由命令 frontmatter 声明支持

---
name: hx-run
hooks:
  - pre
  - post
---

只有命令 frontmatter 声明了 hooks，运行时才会去查找对应的 Hook 文件。

命名规则

按阶段和命令名查找

.hx/hooks/pre_run.md
.hx/hooks/post_run.md
~/.hx/hooks/pre_run.md
~/.hx/hooks/post_run.md
src/hooks/pre_run.md
src/hooks/post_run.md

命名规则见 src/contracts/hook-contract.md。同名 Hook 可在三层同时存在，并按固定顺序串行执行。

启用逻辑:
先加载全局运行规则
→ 读取命令 frontmatter 中的 hooks
→ 对每个已声明阶段查找 pre_<command>.md / post_<command>.md
→ 组装 Hook 链

拼接顺序:
Pre Hook（框架层 → 用户层 → 项目层）
→ 主命令正文
→ Post Hook（项目层 → 用户层 → 框架层）
Project Rules

项目规则

项目规则以 Markdown 文件形式存放在 .hx/rules/ 目录，由 /hx-init 生成骨架，团队手动编辑后永久保留。

<project>/.hx/rules/golden-rules.md   ← 团队黄金原则
<project>/.hx/rules/*.md             ← 其他专项规则文件
初始化

由 /hx-init 生成骨架

/hx-init 分析项目结构后，直接写入项目级规则起始文件到 .hx/rules/。生成的文件分为 hx:auto 区和 hx:manual 区。

手动区永久保留

手动区不会被覆盖

团队在手动区写入的规则内容，运行 /hx-rules update 时不会被覆盖；只有自动区会根据最新模板更新。

更新规则

用 /hx-rules update 更新

框架升级后，可运行 /hx-rules update 刷新自动区内容，同时保留所有手动区定制。团队规则与框架版本解耦。

Troubleshooting

排错

排查问题时优先检查项目根、路径模板、命令解析和 Hook / pipeline 文件位置。

有效入口

正式目录

.hx/config.yaml
.hx/rules/
.hx/pipelines/

项目级配置、项目规则文件和项目级流水线覆盖均应位于此处。

Checklist

检查项

最终生效的是用户级还是项目级配置。

Hook 文件是否按 pre_<command>.md / post_<command>.md 命名。

当前命令 frontmatter 是否声明了 hooks。

paths.requirementDoc、paths.planDoc 与项目实际目录是否一致。

运行 hx doctor 快速检测所有配置问题。
