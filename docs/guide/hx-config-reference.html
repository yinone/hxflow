Precedence

配置入口与优先级

CLI 会合并用户级与项目级配置。

配置文件

两个正式入口

~/.hx/config.yaml
<project>/.hx/config.yaml

前者承载跨项目默认值，后者承载项目级覆盖。

优先级

合并顺序

内置默认值
→ ~/.hx/config.yaml
→ <project>/.hx/config.yaml

项目级覆盖用户级，用户级覆盖框架默认值。

i
合并策略：各字段深合并（deep merge），数组整体覆盖。
Schema

.hx/config.yaml 核心字段

核心字段如下。

object

gate_commands

定义质量门控命令，按 lint → build → type → test 顺序执行。只执行已定义的步骤。

gate_commands:
  lint: npm run lint
  type: npm run type-check
  test: npm run test
object

paths

所有字段均为路径模板，支持 {feature} 和 {taskId} 占位符。其中 {feature} 表示当前需求的内部 feature key。初始化后会显式写出默认值，便于后续编辑。

paths:
  src: app                           # 源码目录（默认 src）
  requirementDoc: docs/req/{feature}.md   # 需求文档模板
  planDoc: docs/plans/{feature}.md        # 执行计划模板
  progressFile: docs/plans/{feature}-progress.json  # 进度文件模板
  taskDoc: 业务线/{feature}/任务/{taskId}/执行.md    # 任务文档模板（可选）
object

hooks

显式声明要注入的 Hook 文件。

hooks:
  run:
    pre:
      - .hx/hooks/extra.md
    post:
      - ~/.hx/hooks/review-extra.md
Paths

路径模板字段

所有 paths 字段均为路径模板，支持 {feature} 和 {taskId} 占位符替换。其中 {feature} 表示当前需求的内部 feature key。下面出现的 docs/requirement 与 docs/plans 仅表示默认值，不是强制目录。

默认值

paths 的默认值（未写入时生效）

requirementDoc → docs/requirement/{feature}.md
planDoc        → docs/plans/{feature}.md
progressFile   → docs/plans/{feature}-progress.json
taskDoc        → （无默认，可选）
src            → src
覆盖方式

推荐显式写出最终值

paths:
  requirementDoc: 需求/{feature}.md
  taskDoc: 任务/{feature}/{taskId}/执行.md

当前实现会在 .hx/config.yaml 中显式写出默认配置和样板入口；CLI 与 CLAUDE.md 注入块都会按最终合并后的 paths 展示真实目录。

!
约定：/hx-init 会自动检测项目目录结构，并把最终生效的 paths 配置显式写入 .hx/config.yaml。
Hooks

Hook 配置

Hook 支持约定式和配置式两种定义方式。

约定式

文件存在即加载

~/.hx/hooks/run-pre.md
~/.hx/hooks/run-post.md
.hx/hooks/run-pre.md
.hx/hooks/run-post.md

配置成本较低。

配置式

显式列出文件路径

hooks:
  doc:
    pre:
      - .hx/hooks/doc-pre.md
  run:
    post:
      - ~/.hx/hooks/style-extra.md

用于精确控制命令粒度。

最终加载顺序:
全局约定 Hook
→ 项目约定 Hook
→ hooks.<command>.<phase> 中声明的数组

输出拼接方式:
Pre Hook
→ 主 Prompt
→ Post Hook
Project Rules

项目规则

项目规则以 Markdown 文件形式存放在 .hx/rules/ 目录，由 /hx-init 生成骨架，团队手动编辑后永久保留。

<project>/.hx/rules/golden-rules.md   ← 团队黄金原则
<project>/.hx/rules/*.md             ← 其他专项规则文件
初始化

由 /hx-init 生成骨架

/hx-init 分析项目结构后，将 src/templates/ 中的规则骨架渲染到 .hx/rules/。生成的文件分为自动区和手动区，以注释分隔。

手动区永久保留

手动区不会被覆盖

团队在手动区写入的规则内容，运行 /hx-rules update 时不会被覆盖；只有自动区会根据最新模板更新。

更新规则

用 /hx-rules update 更新

框架升级后，可运行 /hx-rules update 刷新自动区内容，同时保留所有手动区定制。团队规则与框架版本解耦。

Troubleshooting

排错

排查问题时优先检查配置优先级、Profile 搜索根和文档路径模板。

有效入口

正式目录

.hx/config.yaml
.hx/rules/
.hx/pipelines/

项目级配置、项目规则文件和项目级流水线覆盖均应位于此处。

Checklist

检查项

最终生效的是用户级还是项目级配置。

Hook 文件是否在被查找的根目录中。

paths.requirementDoc、paths.planDoc 与项目实际目录是否一致。

运行 hx doctor 快速检测所有配置问题。
