First Look

命名与职责

区分产品名、命令名和包名。

概念层

产品名与命令名

HXFlow 是产品名，终端命令为 hx，npm 包名为 @hxflow/cli。

使用层

hx 的职责

负责组织文档、计划、项目规则与上下文，并生成执行 Prompt 或运行校验命令。

i
顺序：执行链路 → 运行机制 → 配置参考。
Glossary

术语速读

先把两个高频词读顺，再看命令链路会轻很多。

术语 01

feature

它是“当前需求在 HXFlow 里的内部短 ID”，不是给业务方看的标题。比如“用户登录页支持手机号登录”这类需求，系统可能生成 user-login 作为 key。后续很多文件都会以它命名，所以它更像一个稳定的文件前缀，而不是展示名。

术语 02

.hx/rules/*.md 项目规则文件

它是”本项目执行时遵守的团队规则”。由 /hx-init 生成骨架，存放在 .hx/rules/ 目录。其中 golden-rules.md 包含团队黄金原则；手动区的内容永久保留，不会被 /hx-rules update 覆盖。

i
一句话记忆：feature 用来串起“同一个需求”的文档和进度文件，.hx/rules/*.md 决定“按什么团队规则生成和执行”。正常情况下 feature 自动生成续接，规则文件在项目初始化时配置好，后续只需按需更新。
Install

安装

安装 npm 包后会自动补齐全局入口；需要手动修复时再执行 hx setup。随后在 Claude Code 中运行 /hx-init（Codex 中运行 $hx-init）初始化项目。

终端操作

安装与验证

# 全局安装 CLI
npm install -g @hxflow/cli --registry https://npm.cdfsunrise.com/

# 查看版本
hx version

# 如需手动修复全局资源，可重新执行
hx setup
资源写入

安装链写入目录

~/.hx/settings.yaml
~/.hx/commands/
~/.hx/hooks/
~/.hx/pipelines/
~/.claude/skills/       ← Claude Code 专用入口
~/.agents/skills/       ← 其他 agent 共用入口

运行 hx setup 会补齐这些入口。运行时仍按三层优先级路由到实体 command 文件。

Init

项目初始化（Agent 命令）

在 Claude Code 或 Codex 中运行；自动分析项目结构，生成配置与规则文件骨架。

# 在 Claude Code 中运行（自动分析项目结构）
/hx-init

# 在 Codex 中运行
$hx-init
输出

写入的文件

.hx/config.yaml                 ← 项目级运行时配置
CLAUDE.md / AGENTS.md          ← 注入 harness 标记块
.hx/rules/*.md                 ← 项目规则文件骨架
.hx/commands/README.md         ← 项目级 skill 目录说明
.hx/commands/hx-your-command.md.example
.hx/hooks/README.md            ← Hook 目录说明
.hx/hooks/pre_run.md.example
.hx/hooks/post_run.md.example
.hx/pipelines/default.yaml     ← 项目级流水线样板
约束

项目主入口是 .hx/config.yaml

Agent 会分析技术栈、门控命令和目录结构，显式写入可编辑的默认配置与样板文件，避免用户后续再猜入口位置。

Workflow

执行链路

支持一键自动化（/hx-go）和手动逐步执行两种模式。主路径默认是 /hx-doc → /hx-plan → /hx-run → /hx-qa → /hx-mr。

✓
推荐：优先使用 /hx-go 一键执行主路径。只有在需要断点恢复或手动指定时，才显式传入 feature。
01
创建需求文档
执行 /hx-doc 创建需求文档；命令会根据当前输入或已连接需求来源整理事实，并自动生成项目内稳定的 feature。
02
生成执行计划
补全验收标准后执行 /hx-plan；系统会自动从当前需求文档续接并生成计划 Markdown 和 *-progress.json。
03
执行需求
执行 /hx-run，优先处理依赖已满足的 pending 任务；同一依赖层内若都允许并行，可交给多个子 agent 并发推进。命令会先自动完成当前需求的执行前预检。仅在调试/恢复单个计划任务时才传 --plan-task。
04
质量校验
执行 /hx-qa 运行主路径质量校验。若需要人工审查、失败修复或工程清理，再按需使用 /hx-review、/hx-fix、/hx-clean。
05
交付
执行 /hx-mr 生成 Merge Request 上下文；如果前面已建立最近执行上下文，通常不需要再手工输入 feature。
# 手动逐步执行（Claude）
/hx-doc "用户登录页支持手机号登录"
/hx-plan
/hx-run
/hx-qa
/hx-mr

# 手动逐步执行（Codex）
$hx-doc "用户登录页支持手机号登录"
$hx-plan
$hx-run
$hx-qa
$hx-mr

# 或一键执行（Claude）
/hx-go

# 或一键执行（Codex）
$hx-go

# 从指定步骤恢复（Claude）
/hx-go user-login --from run
✓
可选诊断：若 /hx-run 因输入不完整而停止，可额外执行 /hx-ctx 查看具体缺失项。
Example

完整例子

下面用一个真实心智的例子，把“需求事实 → feature → 文档 → 计划 → 执行 → 交付”串起来。

场景

需求：任务系统里有一个登录相关任务

# 任务来源
任务 ID: 12345
任务标题: 用户登录页支持手机号登录

这时你可以把它理解成：”我要做一个登录需求，执行时按项目 .hx/rules/ 中定义的团队规则运行。”

命令

最常见的两种启动方式

# 一步一步来
/hx-doc "用户登录页支持手机号登录"
/hx-plan
/hx-run
/hx-qa
/hx-mr

# 或者直接一键跑主路径
/hx-go

如果只是第一次体验，优先用这两套命令，不要先手输 feature。

✓
这一步发生了什么：/hx-doc 会读取当前需求详情，整理出需求标题，并自动生成一个项目内稳定的中文 feature。后面整条链路都会围绕这个标识工作。
生成结果

你会在仓库里看到这些文件

docs/
├── requirement/
│   └── user-login.md
└── plans/
    ├── user-login.md
    └── user-login-progress.json

这里的 user-login 就是 feature。它把“需求文档、计划文档、进度文件”三件事绑定成同一个需求上下文。

文件内容

文档头部大概会长这样

# docs/requirement/user-login.md
# 需求：用户登录页支持手机号登录

> 内部主键（Feature Key）：user-login

# docs/plans/user-login-progress.json
{
  "feature": "user-login",
  "requirementDoc": "docs/requirement/user-login.md",
  "planDoc": "docs/plans/user-login.md",
  "createdAt": "2026-04-01T10:00:00+08:00",
  "updatedAt": "2026-04-01T10:00:00+08:00",
  "completedAt": null,
  "lastRun": null,
  "tasks": [
    {
      "id": "TASK-BE-01",
      "name": "新增手机号登录接口",
      "status": "pending",
      "dependsOn": [],
      "parallelizable": false,
      "output": "",
      "startedAt": null,
      "completedAt": null,
      "durationSeconds": null
    }
  ]
}

所以读到这些字段时可以直接这样理解：feature 说明“这是哪个需求”，dependsOn 定义 task 依赖，parallelizable 定义同一依赖层内是否允许并行执行。

i
为什么后面常常不用再写 feature：当你刚执行过 /hx-doc 或当前打开的需求文档头部已经写着 Feature: user-login，/hx-plan、/hx-run、/hx-mr 一般都能自动续接到这个需求。
i
什么时候才需要手动写 key：通常只在断点恢复、同一项目里同时有多个需求、或者自动定位失败时。例如：/hx-go user-login --from run。
Pitfalls

常见问题

以下错误会直接影响执行结果或上下文一致性。

错误 01

执行了 hx plan，却没先补需求文档

计划直接依赖当前需求文档内容。不完整的 AC 会被固化到计划和进度文件。

错误 02

误将 /hx-run 视为代码执行器

/hx-run 是 Agent 命令，由 Claude Code 或 Codex 读取并执行，不是普通 Shell 命令行工具。

错误 03

安装链没有跑通，或没有运行 hx-init

装包阶段会自动初始化全局入口与目录结构；hx-init 初始化项目配置。如果缺少全局入口，优先重装包或手动执行 hx setup 修复。

错误 04

将 hx setup 和 /hx-init 搞混

hx setup 是机器级修复入口；/hx-init 是每个新项目都要执行的项目级初始化。

错误 05

误以为还要手动执行收尾命令

/hx-run 会自动回写 *-progress.json。不要再手工维护任务完成状态，除非明确在做调试或恢复。
