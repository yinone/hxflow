First Look

命名与职责

区分产品名、命令名和包名。

概念层

产品名与命令名

HXFlow 是产品名，终端命令为 hx，npm 包名为 @hxflow/cli。

使用层

hx 的职责

负责组织文档、计划、Profile 规则与上下文，并生成执行 Prompt 或运行校验命令。

i
顺序：执行链路 → 运行机制 → 配置参考。
Glossary

术语速读

先把两个高频词读顺，再看命令链路会轻很多。

术语 01

feature key

它是“当前需求在 HXFlow 里的内部短 ID”，不是给业务方看的标题。比如“用户登录页支持手机号登录”这类需求，系统可能生成 user-login 作为 key。后续很多文件都会以它命名，所以它更像一个稳定的文件前缀，而不是展示名。

术语 02

profile

它是“本次执行采用哪套团队规则”。里面会定义模板、架构层级、质量门控命令和默认路径。最常见的是 base，它表示先按框架内置最小规则跑通；如果团队有定制流程，再换成自己的 profile。

i
一句话记忆：feature key 用来串起“同一个需求”的文档和进度文件，profile 用来决定“按什么规则生成和执行”。正常情况下你通常只需要显式写 --profile，不需要手写 feature key。
Install

安装

安装 npm 包后执行 hx setup 初始化全局资源，然后在 Claude Code 或 Codex 中运行 hx-init 初始化项目。

终端操作

安装与验证

# 全局安装 CLI
npm install -g @hxflow/cli --registry https://npm.cdfsunrise.com/

# 查看版本
hx version

# 初始化全局资源（仅需运行一次）
hx setup
资源写入

hx setup 写入目录

~/.hx/config.yaml
~/.hx/commands/
~/.hx/profiles/
~/.hx/pipelines/
~/.claude/commands/    ← Claude 转发器
~/.codex/skills/hxflow/ ← Codex skill bundle

这些入口都不含业务逻辑，运行时按三层优先级路由到实体命令文件；skill 仍由 Agent 自身机制管理。

Init

项目初始化（Agent 命令）

在 Claude Code 或 Codex 中运行 hx-init；可选 --profile 指定要使用的 Profile 名称。框架默认内置 base。

# 在 Claude Code 中运行（自动分析项目结构）
/hx-init

# 在 Codex 中运行
hx-init

# 或显式指定 profile
/hx-init --profile base
输出

写入的文件

.hx/config.yaml            ← 显式默认配置 + paths
CLAUDE.md                  ← 注入 harness 标记块
.hx/commands/*.example     ← 自定义命令样板
.hx/hooks/*.example        ← Hook 样板
.hx/pipelines/default.yaml ← 项目级流水线样板
.hx/profiles/<name>/       ← 自定义 profile（如需）
约束

项目主入口是 .hx/config.yaml

Agent 会分析技术栈、门控命令和目录结构，显式写入可编辑的默认配置与样板文件，避免用户后续再猜入口位置。

Workflow

执行链路

支持一键自动化（/hx-go）和手动逐步执行两种模式。主路径默认是 /hx-doc → /hx-plan → /hx-run → /hx-qa → /hx-mr。

✓
推荐：优先使用 /hx-go --task 12345 --profile base 一键执行主路径。只有在需要断点恢复或手动指定时，才显式传入 feature key。
01
创建需求文档
执行 /hx-doc --profile base 创建需求文档；可用 --task <id> 直接从任务系统拉取需求来源。feature key 由系统自动生成，正常情况下不用关心。
02
生成执行计划
补全验收标准后执行 /hx-plan --profile base；系统会自动从当前需求文档续接并生成计划 Markdown 和 *-progress.json。
03
执行需求
执行 /hx-run --profile base，按计划顺序完成当前需求下的全部 pending 任务；命令会先自动完成当前需求的执行前预检。仅在调试/恢复时才传 --task。
04
质量校验
执行 /hx-qa --profile base 运行主路径质量校验。若需要人工审查、失败修复或工程清理，再按需使用 /hx-review、/hx-fix、/hx-clean。
05
交付
执行 /hx-mr 生成 Merge Request 上下文；如果前面已建立最近执行上下文，通常不需要再手工输入 feature key。
# 手动逐步执行
/hx-doc --task 12345 --profile base
/hx-plan --profile base
/hx-run --profile base
/hx-qa --profile base
/hx-mr

# 或一键执行
/hx-go --task 12345 --profile base

# 从指定步骤恢复
/hx-go user-login --from run --profile base
✓
可选诊断：若 /hx-run 因输入不完整而停止，可额外执行 /hx-ctx --profile base 查看具体缺失项。
Example

完整例子

下面用一个真实心智的例子，把“任务来源 → feature key → 文档 → 计划 → 执行 → 交付”串起来。

场景

需求：任务系统里有一个登录相关任务

# 任务来源
任务 ID: 12345
任务标题: 用户登录页支持手机号登录

# 我们先用最简单的规则集
profile: base

这时你可以把它理解成：“我要做一个登录需求，并且这次先按 base 这套默认规则来执行。”

命令

最常见的两种启动方式

# 一步一步来
/hx-doc --task 12345 --profile base
/hx-plan --profile base
/hx-run --profile base
/hx-qa --profile base
/hx-mr

# 或者直接一键跑主路径
/hx-go --task 12345 --profile base

如果只是第一次体验，优先用这两套命令，不要先手输 feature key。

✓
这一步发生了什么：/hx-doc 会读取任务 12345 的标题和描述，再自动生成一个适合做文件名的 kebab-case key。假设它生成的是 user-login，后面整条链路就会围绕这个 key 工作。
生成结果

你会在仓库里看到这些文件

docs/
├── requirement/
│   └── user-login.md
└── plans/
    ├── user-login.md
    └── user-login-progress.json

这里的 user-login 就是 feature key。它把“需求文档、计划文档、进度文件”三件事绑定成同一个需求上下文。

文件内容

文档头部大概会长这样

# docs/requirement/user-login.md
# 需求：用户登录页支持手机号登录

> 内部主键（Feature Key）：user-login
> Profile：base

# docs/plans/user-login-progress.json
{
  "feature": "user-login",
  "profile": "base",
  "tasks": [
    { "id": "TASK-BE-01", "status": "pending" }
  ]
}

所以读到这些字段时可以直接这样理解：feature 说明“这是哪个需求”，profile 说明“按哪套规则拆任务和做门控”。

i
为什么后面常常不用再写 key：当你刚执行过 /hx-doc 或当前打开的需求文档头部已经写着 Feature Key: user-login，/hx-plan、/hx-run、/hx-mr 一般都能自动续接到这个需求。
i
什么时候才需要手动写 key：通常只在断点恢复、同一项目里同时有多个需求、或者自动定位失败时。例如：/hx-go user-login --from run --profile base。
Pitfalls

常见问题

以下错误会直接影响执行结果或上下文一致性。

错误 01

执行了 hx plan，却没先补需求文档

计划直接依赖当前需求文档内容。不完整的 AC 会被固化到计划和进度文件。

错误 02

误将 /hx-run 视为代码执行器

/hx-run 是 Agent 命令，由 Claude Code 或 Codex 读取并执行，不是普通 Shell 命令行工具。

错误 03

没有先执行 hx setup，或没有运行 hx-init

hx setup 初始化全局入口与目录结构；hx-init 初始化项目配置。两步都必须完成。

错误 04

将 hx setup 和 /hx-init 搞混

hx setup 只需运行一次（机器级），/hx-init 在每个新项目运行（项目级）。

错误 05

误以为还要手动执行收尾命令

/hx-run 会自动回写 *-progress.json。不要再手工维护任务完成状态，除非明确在做调试或恢复。
