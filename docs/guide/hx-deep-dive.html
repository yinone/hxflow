Phases

工作流阶段

CLI 以“主路径 + 辅助能力”组织能力，而不是把所有命令都视为同等级阶段。

Core 01-02

准备输入

/hx-doc、/hx-plan 负责建立需求文档、计划和 progress 状态；/hx-status 负责观察当前链路状态。

Core 03-04

执行与校验

/hx-run 默认执行整个需求，/hx-check 完成主路径核心检查。

Auxiliary

辅助能力

/hx-fix 与 /hx-mr 在检查失败恢复或交付整理时启用；预检、审查、质量门和工程扫描都已收口到 /hx-check。

Runtime

运行时路径

框架将资源划分为系统层、用户层和项目层，Claude 与 Codex 都按同一套优先级路由。

层级 典型路径 角色
FRAMEWORK_ROOT
src/commands/（内置 skill 实体）
src/pipelines/（内置流水线）
 框架内置资源（只读），通过装包或重装同步。skill 不直接在此运行，由入口文件路由。
USER_HX_DIR
~/.hx/commands/（用户 skill 覆盖）
~/.hx/hooks/（用户 Hook）
~/.hx/pipelines/（用户流水线）
~/.hx/settings.yaml
 跨项目共享的用户级覆盖与安装设置。由 hx setup 创建目录结构，用户自行写入覆盖文件。
PROJECT_ROOT
.hx/commands/（项目 skill 覆盖）
.hx/rules/（项目规则文件）
.hx/pipelines/（项目流水线）
.hx/config.yaml
 项目专属覆盖，优先级最高。由 /hx-init 初始化 .hx/config.yaml 与 .hx/rules/。
i
配置边界：~/.hx/settings.yaml 只保存用户级安装设置；项目运行时配置以 .hx/config.yaml 为准。所有命令先继承 src/contracts/runtime-contract.md，再按 src/contracts/resolution-contract.md 解析 commands / pipelines，并按固定顺序执行 hooks。
i
入口机制：Claude Code 通过 ~/.claude/skills/ 运行；其他 agent 统一通过 ~/.agents/skills/ 运行。最终都按项目层 → 用户层 → 系统层的顺序读取同一套实体文件。
Rules Runtime

规则运行时

项目规则以静态文件方式存储在 .hx/rules/ 目录，由 /hx-init 生成骨架，Agent 命令在执行时直接读取。无运行时查找链，无继承逻辑。

规则入口

项目规则文件位置

<project>/.hx/rules/golden-rules.md
<project>/.hx/rules/*.md（其他规则文件）
<project>/.hx/config.yaml（硬变量配置）

所有规则文件由团队手动维护，直接生效，无需 merge 或继承。

初始化与更新

规则文件的生命周期

/hx-init      → 生成 .hx/rules/*.md 骨架
手动编辑      → 写入手动区（永久保留）
/hx-rules update → 更新自动区，不覆盖手动区

手动区与自动区由注释分隔，更新只触及自动区。

i
设计原则：运行时事实收敛为 .hx/config.yaml（硬变量）与 .hx/rules/*.md（规则文件），静态读取，无动态推断。
Context

上下文装配

/hx-run 通过固定清单组装上下文与 Hook，并围绕当前需求的 feature 解析所有路径模板。

主上下文

/hx-run 默认读取的文件

.hx/rules/golden-rules.md
.hx/rules/*.md（其他项目规则文件）
.hx/config.yaml（硬变量配置）
paths.planDoc 对应文件
paths.requirementDoc 对应文件
paths.progressFile 对应文件

路径由 .hx/config.yaml 的 paths.* 模板与当前需求的 feature 决定。

Hook 注入

由 frontmatter 启用，按命名规则查找

---
name: hx-run
hooks:
  - pre
  - post
---

只有命令 frontmatter 声明了 hooks，运行时才会按 pre_run.md / post_run.md 组装 Hook 链。pre 顺序为框架层 → 用户层 → 项目层，post 顺序相反。主路径里像 hx-doc、hx-plan、hx-run 的强制复查步骤属于命令正文的一部分，不依赖项目 Hook 是否存在。

Quality

质量控制

质量控制由主路径校验和按需辅助能力两部分组成。

统一检查

/hx-check

统一承接审查、质量门和工程卫生扫描。可用 --scope review|qa|clean|all 限定范围。

恢复修复

/hx-fix

读取最近一次失败上下文、日志或文件范围，按项目规则修复问题并准备重新进入 /hx-run 或 /hx-check。

质量门

qa scope

/hx-check --scope qa 按固定顺序执行 lint → build → type → test，只跑 .hx/config.yaml 中实际定义过的步骤，遇到失败立即停止。

!
占位符参数：若门控命令包含 {scheme} 等占位符，必须通过 CLI 参数显式传入。
Progress

计划与进度

多个命令依赖 paths.progressFile 模板对应的 progress JSON 进行状态推断；文件同时承载 task 依赖与并行调度元数据。

计划生成

/hx-plan 输出两份文件

输出路径由 .hx/config.yaml 的 paths.planDoc 和 paths.progressFile 模板决定。

状态流转

任务状态与调度字段

pending
in-progress
done
dependsOn: string[]
parallelizable: boolean

任务状态默认由 /hx-run 自动更新；只有 dependsOn 全部满足的 pending task 才算 runnable。同一依赖层内若都允许并行，才可交给多个子 agent 并发处理。

Pipeline

流水线编排

/hx-go 由流水线 YAML 驱动，支持三层覆盖和断点恢复；默认项目内置主路径是 doc -> plan -> run -> check -> mr。

查找顺序

Pipeline YAML 三层路径

<project>/.hx/pipelines/default.yaml
~/.hx/pipelines/default.yaml
框架内置 src/pipelines/default.yaml

找到第一个存在的文件即使用，可按层级覆盖系统默认流水线。

步骤定义

YAML 步骤字段

steps:
  - id: doc
    phase: Phase 01
    name: 需求文档
    command: hx-doc
    checkpoint:
      message: 确认后继续
  - id: run
    command: hx-run
    on_fail: stop

checkpoint 暂停等待用户确认；on_fail: stop 失败时终止流水线。若团队需要，也可以在项目级 pipeline 中显式插入 review 或 clean。

✓
断点恢复：使用 /hx-go user-login --from run 从指定步骤继续，跳过已完成的 doc/plan 阶段。
