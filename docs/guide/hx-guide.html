HXFLOW · OVERVIEW

HXFlow

HXFlow 是需求交付框架。hx CLI 负责安装与维护，所有工作流命令（/hx-*）在 Claude Code 中直接执行。npm 包名为 @hxflow/cli。

Positioning

产品定位

HXFlow 是工程输入、团队规则与质量门控的统一入口。核心理念：工程师通过结构化文档 + Agent 执行的方式交付，而非直接写业务代码。

输入治理

结构化工程输入

通过 /hx-doc → /hx-plan → /hx-run 固化需求、任务边界和执行闭环。/hx-go 支持一键全流程自动化。

团队治理

统一团队差异

通过 .hx/rules/*.md 项目规则和 .hx/config.yaml 硬变量配置团队约束与门控命令。所有命令先继承全局运行规则，再按项目 → 用户 → 系统解析 commands / pipelines，并以中间件链方式叠加 hooks。

质量闭环

形成质量闭环

/hx-check、/hx-fix 与 /hx-mr 负责主路径收口和恢复闭环。

i
两类命令：hx CLI 只保留 setup、migrate 和 version；/hx-* / $hx-* 是统一的 Agent skill 契约，由 agent 的 skill 入口路由到框架实体执行。
Glossary

先懂两个词

多数人第一次卡住的不是命令，而是不知道 feature 和项目规则分别在框架里扮演什么角色。

术语 01

feature = 当前需求的项目内标识

可以把它理解成“这次需求在 HXFlow 里的文件名前缀”。例如登录需求可能被生成成 user-login，随后需求文档、计划文档、进度文件通常都会围绕它命名：docs/requirement/user-login.md、docs/plans/user-login.md、docs/plans/user-login-progress.json。

术语 02

.hx/rules/*.md = 项目规则文件

项目规则文件由 /hx-init 生成骨架，存放在 .hx/rules/ 目录下。其中 golden-rules.md 定义团队黄金原则，其他规则文件按需添加。规则一经手动区写入，后续 /hx-rules update 不会覆盖。

i
一句话记忆：feature 回答“这是哪一个需求”，.hx/rules/*.md 决定“团队按什么规则执行”。正常主路径里，feature 通常自动生成并自动续接，规则文件在项目初始化时一次性配置好。
Workflow

核心链路

标准链路为“准备输入 → 执行需求 → 核心检查 → 交付”。可使用 /hx-go 一键驱动，也可逐步手动控制；/hx-check 统一承接预检、审查、质量门和工程卫生扫描，/hx-fix 负责失败恢复。

1
项目初始化
/hx-init 分析项目结构，写入 .hx/config.yaml 并在 CLAUDE.md 注入上下文标记块。每个项目只需运行一次。
2
需求文档
/hx-doc 创建需求文档；命令会基于当前需求上下文整理需求内容并生成项目内稳定的 feature。产出后会再新开一个子 agent 复查需求事实、验收标准和头部元信息，主 agent 修正后再输出。
3
执行计划
/hx-plan [<feature>] 生成任务拆分和 progress JSON；若省略 feature，会优先从当前需求文档自动续接。计划和进度初始化后会再新开一个子 agent 复查 TASK 拆分、验收标准以及 dependsOn / parallelizable 是否合理。
4
任务执行
/hx-run [<feature>] [--plan-task <TASK-ID>] 默认先按 task 依赖关系选择 runnable task；同一依赖层内若都允许并行，可交给多个子 agent 并行处理，否则按计划顺序串行执行。执行结束后还会再新开一个质量复查子 agent 主动寻找边界条件、异常分支和回归风险。只在调试或恢复时才传 --plan-task 下钻到单任务。
5
核心检查
/hx-check 统一执行预检、审查、质量门和工程卫生扫描。存在失败项时进入 /hx-fix；全部通过后继续交付。
6
交付与沉淀
/hx-mr 汇总 MR 上下文并输出交付物；交付前如需再次确认质量，只需重复运行 /hx-check。
i
一键模式：/hx-go [<feature>] 读取 pipeline YAML 驱动默认主路径（doc -> plan -> run -> check -> mr），支持 --from <step-id> 从指定阶段恢复；未显式传入 --from 时，会按耐久产物自动判断最早未完成 step。
✓
读图方式：如果你只想先跑通一遍，不要先纠结 feature。直接从 /hx-doc 开始。只有在断点恢复、冲突消歧或手工指定需求时，才需要显式输入。

Constraints

关键约束

以下约束决定框架的实际行为。

命令真源

入口和分发以 bin/hx.js 为准

hx CLI 只暴露 3 个内置命令（setup / migrate / version）；/hx-* 工作流命令由 agent 的 skills 入口路由到 src/commands/ 实体文件执行。

bin/hx.js src/commands/ ~/.claude/skills/ ~/.agents/skills/
配置真源

三层配置有固定解析顺序

项目层（.hx/）→ 用户层（~/.hx/）→ 系统层（框架内置）。运行时规则入口是 src/contracts/runtime-contract.md；实体解析顺序统一看 src/contracts/resolution-contract.md；hooks 按固定顺序串行执行；~/.hx/settings.yaml 只保存安装级设置。
