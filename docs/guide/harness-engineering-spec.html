关于 Harness

什么是 Harness Engineering

Harness Engineering 是一套以 AI Agent 为核心执行者的软件工程方法论。在本仓库中，它被实现为一个 project-rules-driven workflow framework。工程师不再直接编写业务代码，而是通过结构化文档、架构约束和自动化流水线驱动 Agent 产出高质量代码。

为什么需要 Harness
传统开发中，AI 辅助编码（Copilot 模式）仅解决了「写代码」环节。但软件工程的核心挑战在于：需求理解、架构一致性、质量保障和团队协作。Harness 将这些环节编码为可复用的规则和流水线，确保 Agent 在确定性约束下运行。
核心理念
工程师负责设计约束、编写规范、定义验收标准，Agent 在确定性的框架内执行。输出物是可追溯、可复现的结构化产物，而非依赖个人经验的临时决策。

Phase 01

需求接入与意图锁定

需求统一从公司协作平台（DevOps）获取，通过 MCP 拉取任务详情后转化为结构化文档。口头需求、会议讨论、模糊描述不能直接驱动 Agent。

PHASE 01
需求
格式化
全员
⏱ 0.5 ~ 1h / 需求

每一条需求都必须产出一个结构化文档，写入 paths.requirementDoc 对应位置（默认在 docs/requirement/）。需求来源通常是公司 DevOps 协作平台；优先通过 /hx-doc 读取当前输入或已连接来源中的需求事实，再转化为 Agent 可执行的标准格式。

必须 建议 说明
  • /hx-doc 应从当前输入或已连接需求来源中获取标题、描述、优先级、关联需求等事实，并在内部生成项目内唯一、可复用的 feature，作为需求文档的关联标识
  • 以 Markdown 写需求文档，文件命名遵循 paths.requirementDoc 模板；若使用默认值，则为 docs/requirement/[feature].md。文档头部记录来源信息与 feature 以保持双向追溯
  • 明确写出「验收标准（AC）」，每条 AC 必须具体到可以被自动化测试验证的行为，而不是主观描述
  • 标明本需求明确的边界约束（不做什么）
  • 在文档顶部标注依赖的其他需求或模块，便于 Agent 自动拉取关联上下文
  • 同步更新执行计划与进度文件，让后续命令可以直接从文档和状态文件恢复上下文
需求文档模板 · 默认路径示例：docs/requirement/[feature].md
# 需求：用户登录接口
> 内部主键（Feature Key）：user-login

## 来源
DevOps 任务: #12345 — 用户登录接口开发
优先级: P1｜迭代: Sprint 2024-03

## 背景
用户需要通过邮箱+密码登录，获取 JWT Token

## 验收标准（AC）
- AC-001: POST /api/auth/login 接收 { email, password }，返回 { token, expiresAt }
- AC-002: 密码错误返回 401，body: { code: "INVALID_CREDENTIALS" }
- AC-003: 连续失败 5 次，账号锁定 15 分钟，返回 429
- AC-004: Token 有效期 24h，支持 refresh
- AC-005: 登录成功写入结构化日志，字段含 userId、ip、userAgent

## 影响层级
Service（认证逻辑）/ Repo（用户查询）/ Runtime（Token 签发）

## 边界约束（不做）
- 本期不做 OAuth 第三方登录
- 不做短信验证码登录

## 依赖
- docs/requirement/user-schema.md（用户数据结构）

Phase 05

代码评审

评审聚焦架构合规性与 AI 特征性问题，行级规则由 CI 自动检查与阻断。

PHASE 05
代码
评审
全员
⏱ 20 ~ 40min / PR

AI 生成代码具有特定的失效模式，Review 清单针对这些特征设计。

▸ 前端 Review 清单
  • Props 接口是否和需求文档一致——Agent 有时会自行扩展或收缩接口
  • 是否绕过了现有 UI 组件库——Agent 容易「重新发明」已有的 Button / Input，产生不一致
  • 状态管理是否符合约定——检查是否意外引入了新的 global state 或绕过了约定的 Context
  • 过度抽象检查：Agent 有时会对只用一次的逻辑创建没必要的 Hook 或 HOC
  • 无障碍（a11y）基础检查：表单是否有 label、图片是否有 alt、交互元素是否可键盘操作
  • 不需要逐行审查 CSS / 样式实现细节，由截图或 Storybook 视觉验证即可
▸ 后端 Review 清单
  • 架构层级合规——Service 层是否直接导入了 Runtime/UI 模块？Repo 层是否包含了业务逻辑？
  • 错误码是否使用了已定义的体系——禁止出现未在 error-codes.md 中定义的错误码或新的 HTTP 状态码组合
  • 日志字段是否完整——结构化日志中必须包含约定字段（userId、action、ip 等），Lint 会自动检查
  • 过度错误处理检查：Agent 有时会为正常业务路径添加多余的 try-catch，增加维护成本
  • 文档漂移检查：新增的接口参数是否已同步更新到对应需求文档（由 paths.requirementDoc 定位）
  • 单元测试通过 CI 自动验证，Review 中不需要人工计算覆盖率，只检查关键路径是否有测试
▸ 移动端 Review 清单
  • Domain 层是否纯逻辑——不引入 UIKit/SwiftUI/Android SDK/ArkUI 等平台 API（GP-MB-001）
  • 层级依赖合规——UI 层是否直接调用了 Data 层（必须经过 Presentation）（GP-MB-002）
  • 内存安全——ViewModel 是否持有 View 强引用（GP-MB-007），是否有循环引用风险
  • 线程安全：UI 更新是否在主线程（GP-MB-004），后台任务是否正确切换
  • 平台特定：iOS 无 force unwrap、Android 无 !!、HarmonyOS 无 FA 模型废弃 API
  • 深色模式、横竖屏切换、无障碍（VoiceOver/TalkBack）由 QA 阶段在真机验证
两个全员必检项：① 变更是否破坏了现有的 Lint 规则（CI 自动阻断，但要理解原因）；② 文档与代码是否一致——这是 Agent 下次执行质量的基础，不同步就是技术债。

Skill 编写规范

Skill 是把团队隐性知识编码为 Agent 可复用模块的机制。写得好，Agent 每次精准触发、正确执行；写得差，Agent 视而不见或走歪。本章覆盖结构、触发描述、内容原则与迭代方法。

一句话定义：Skill = 由 agent 自身机制加载的能力目录，包含一个 SKILL.md 文件以及可选的脚本、参考文档和资产文件。Skill 可安装在 .claude/skills/、~/.claude/skills/， 或以软链方式指向共享仓库 ~/qiyuan-common-skills/skills/。Agent 每次会话读取所有 Skill 的 frontmatter（name + description），决定是否调用某个 Skill 的完整内容。
Skill 安装与管理
Skill 由 Agent 自身机制安装和发现，不属于 hx setup / hx upgrade 的职责。Skill 可以安装在两个级别：项目级（仅当前项目可用）和用户级（所有项目共享）。选择取决于 Skill 的通用程度。
P
Project 级别
.claude/skills/
仅当前项目可用，随代码库一起版本管理。适合团队专属的工作流、架构规范、发布流程等。
  • 跟随 git 提交，团队成员自动获得
  • 可在 .claude/skills/ 目录下按团队组织子目录
  • 优先级高于同名 User 级 Skill
U
User 级别
~/.claude/skills/
所有项目共享的个人 Skill。适合跨项目通用的工具：代码审查习惯、个人提效脚本、通用模板等。
  • 不随项目版本管理，仅本地存在
  • 所有项目会话都会加载 frontmatter
  • 同名时被 Project 级覆盖

SKILL.md必须
含 YAML frontmatter（name、description）和 Markdown 正文。正文是 Agent 触发后读取的详细指令。控制在 500 行以内，超出则拆分到 references/ 并用指针引用。

可执行脚本，用于确定性的重复操作。Agent 可直接调用脚本而无需加载内容到上下文，节省 token。
references/可选
按需加载的参考文档。SKILL.md 中用条件指针告知 Agent 何时读哪个文件：「当目标平台为 AWS 时，阅读 references/aws.md」。
assets/可选
输出时用到的文件：模板、图标、字体等。SKILL.md 中写明路径，Agent 执行时直接引用，无需描述文件内容。
三级渐进加载
LEVEL 1 · 始终在上下文
frontmatter（name + description）
每次会话启动时，所有 Skill 的 frontmatter 都被载入。Agent 依赖 description 决定是否触发。这是最关键的部分，写得好不好直接决定触发率。
⌀ ~100 词，保持精简
LEVEL 2 · 触发后加载
SKILL.md 正文
Agent 决定使用某个 Skill 后，完整读取正文。包含执行步骤、注意事项、格式要求和指针引用。超过 500 行应拆分，多余内容放 references/。
⌀ 理想 <500 行
LEVEL 3 · 按需加载
scripts / references / assets
只在正文明确指向时加载。大型参考文档（>300 行）需在文件顶部提供目录。脚本通常直接执行，不加载内容，节省上下文空间。
无大小限制
description 写法——触发的核心

Agent 有「低触发」倾向——在你认为应该触发的场景下，它往往不触发。description 需要主动、明确地列出触发条件，比你以为的更宽。

✗ 低质量 description
description: | How to deploy services to the cloud. Covers AWS, GCP, and Azure deployment workflows and best practices.
✓ 高质量 description
description: | Cloud deployment for AWS, GCP, Azure. Use whenever user mentions deploying, hosting, releasing, publishing, shipping any service or container—even without the word "cloud". Also trigger for CI/CD config, production env vars, or simply "how do I get this live". Not for local dev setup.
▸ 规则 1
做什么 + 何时触发，缺一不可
「做什么」让 Agent 知道能力范围；「何时触发」列举用户会说的词语和隐含场景（「帮我上线」不说「部署」，但应触发部署 Skill）。两者都要写。
▸ 规则 2
触发条件要比你以为的更宽
列举同义词、近义场景、用户的非正式说法。宁可稍宽不可太窄——Agent 读完 SKILL.md 后如发现不适用会自行放弃，不触发才是真正的损失。
▸ 规则 3
执行规则不放进 description
description 只负责触发时机，执行规则、格式要求、注意事项一律放进正文。description 过长稀释触发精度，也浪费每次会话的上下文空间。
▸ 规则 4
简单任务不会触发 Skill
Agent 对单步任务（「读文件」「列表格」）不触发 Skill——它能直接处理。Skill 的目标是复杂的、多步的、专业化的任务，description 要体现这种复杂性。
SKILL.md 正文编写原则
P-01
最小惊讶原则
Agent 产出应和「了解这个领域的高级工程师」的期望完全一致。如果你看到产出心想「啊，不对」——说明 SKILL.md 描述得不够精确，需要补充。
P-02
步骤而非描述
写「先做 A，再做 B，然后做 C」，不要写「这个 Skill 能帮你做 A、B、C」。正文是执行指令，Agent 需要的是动词开头的、明确的动作序列。
P-03
明确输入输出
每个 Skill 都要有清晰产物。「输出到 outputs/[name].md 并调用 present_files」比「生成一个文档」好。模糊的输出描述导致产出形式不稳定。
P-04
用指针替代内联
大块参考内容不要内联进 SKILL.md，放进 references/ 并用条件指针引用：「如果目标是 React，先阅读 references/react.md」。
P-05
显式写出反例
对容易误判的情况，直接写「不适用场景」。反例和正例同样重要，能大幅减少误触发和产出方向偏差。
P-06
团队知识编码化
每次有人说「这种情况应该这样处理」——就是写 Skill 或更新 Skill 的信号。Skill 是把资深工程师脑中隐性知识变成任何人都能获取的显性规则的机制。

提示词最佳实践

提示词是工程师与 Agent 之间的唯一接口。写得清晰，Agent 产出稳定；写得模糊，每次结果都像在开盲盒。这里提炼了在 Harness 工作流中反复验证有效的写法规律。

核心原则：描述「期望的结果状态」，而不是「实现步骤」。给 Agent 足够的约束（引用文档、指定路径、锁定接口），但不要替它规划每一行代码。约束是边界，不是剧本。
提示词解剖——六个构成要素

一条高质量提示词通常由以下要素组合而成。不是每条都必须有六个，但越复杂的任务越需要覆盖更多要素。

① 角色 / 模式
「按照 docs/plans/auth-login-api.md 中的 TASK-BE-03 执行」
定位上下文。告诉 Agent 它在整个任务体系中的位置，而非从零开始理解。引用执行计划 ID 是最高效的方式。
② 目标
「创建 src/service/authService.ts，实现 login 方法，满足 AC-003 的失败限制逻辑」
一条提示词只有一个核心目标。多个目标拆成多次执行，每次一个产物。
③ 上下文引用
「入参/出参类型使用 src/types/auth.ts 中已定义的 LoginPayload / LoginResponse」
把 Agent 需要的信息指向仓库中已有的文档和类型，而非在提示词中重复描述。引用 > 内联。
④ 约束边界
「通过 src/repo/userRepo.findByEmail() 查用户，禁止在 Service 层直接操作数据库」
写清楚「不能做什么」和「必须用什么」。负面约束（禁止…）和正面约束（必须用…）同样重要。
⑤ 验收动作
「完成后运行 /hx-qa，所有检查通过后开 PR，并更新进度文件中 TASK-BE-03 状态为完成」
告诉 Agent 「完成」的标准是什么，以及完成后做什么。没有这一步，Agent 不知道何时停止。
⑥ 思考提示
「如果对失败计数的存储位置有疑问，先阅读 docs/requirement/auth-system.md 的「会话状态」章节再决定」
对复杂决策点，提前给出「遇到歧义时去哪里找答案」，而非让 Agent 自行猜测或停下来问你。

上下文预算——如何分配有限的上下文窗口

上下文窗口利用率超过 40% 时，Agent 性能开始下降。同一个会话中累积的历史越多，当前任务越容易跑偏。以下是在 Harness 工作流中管理上下文的实践规则。

规则 01
一个会话，一个 TASK
每个 TASK-XX 开启新会话执行。不要在同一会话中连续执行 TASK-01、TASK-02、TASK-03——历史上下文会干扰后续任务，且错误会累积传播。
规则 02
引用替代粘贴
用文件路径引用（「见 src/types/auth.ts」）替代直接粘贴内容。Agent 会在需要时自行读取，比你粘贴全文更节省上下文，也更准确（读到的是最新版本）。
规则 03
主上下文保持精简
不要再维护额外的索引文件。让 `.hx/config.yaml`、需求文档、执行计划和 progress JSON 成为唯一上下文入口，避免多一份需要同步的状态。
规则 04
长日志截取关键段
粘贴日志时，只粘贴报错核心段落（错误类型、堆栈顶部 5 行、关键上下文行）。完整日志文件保存到磁盘并告知路径，让 Agent 按需读取，而非一次性塞入提示词。
规则 05
复杂任务先规划再执行
对 5 个子任务以上的需求，先单独开一个会话让 Agent 生成执行计划（输出到 paths.planDoc 对应位置，默认在 docs/plans/），再分别开会话执行每个子任务。规划会话和执行会话分离。
规则 06
失败后重开会话
Agent 在同一会话中失败 2 次以上，不要继续追加修正指令。失败积累会污染上下文。正确做法：修复文档或约束，重开干净会话重新执行。
高阶技巧——提升产出质量
✗ 会降低质量的写法
「你是一个资深工程师，请帮我……」 // 空洞的角色设定，Agent 忽略 「请尽可能详细地……」 // 模糊的质量要求，没有约束力 「帮我写好一点」 // Agent 无法量化「好一点」 「先做 A，再做 B，然后 C……」 // 剧本式提示词，压制 Agent 判断力 「你觉得最好的方式是什么？」 // 开放式提问，产出不可预期
✓ 会提升质量的写法
「按照 TASK-BE-03，完成后 /hx-qa 通过」 // 具体任务 + 具体验收，可复现 「满足 AC-001、AC-002、AC-003」 // 引用验收标准，Agent 逐条对齐 「文件 ≤ 400 行，无 any，测试覆盖率 ≥ 80%」 // 可量化的约束，Agent 能自验 「如果对 X 有疑问，先读对应需求文档（例如默认路径下的 docs/requirement/xxx.md）」 // 预埋决策入口，减少来回确认 「不允许新增外部依赖，只用 Node.js 内置模块」 // 负面约束，缩小方案空间
▸ 技巧 A
让 Agent 先说计划再执行
对复杂任务在提示词末尾加「先简述执行计划（3~5 点），确认后再开始实现」。这使 Agent 显式化推理过程，便于在执行前发现方向偏差。
▸ 技巧 B
用 AC 代替叙述式需求
「返回 { token, expiresAt }」这种叙述不如「满足 AC-001：POST /api/auth/login 返回 { token: string, expiresAt: Date }」。AC 是可验证的合约，叙述是可解释的描述，两者精确度不同。
▸ 技巧 C
提供反例比提供正例更有效
「不要用 console.log，不要 throw new Error，不要在 Service 层直接 import 数据库客户端」比「请遵循最佳实践」效果好十倍。反例约束边界，正例描述方向；边界比方向更精确。
▸ 技巧 D
失败时分析环境，而非追加指令
Agent 产出不符合预期时，先问自己「是文档缺了什么、类型定义不够清晰、还是约束没写清楚」，修复后重开会话。在失败的会话里追加「不对，重新来」会让上下文越来越乱。

Claude Code 使用指南

Claude Code 不是 IDE 插件，而是一个原生 Linux 命令行工具。它直接运行在终端中，像 git、npm 一样成为工程师工具链的一部分。这意味着它天然继承了 Unix 哲学：可组合、可脚本化、可自动化。

为什么选择 Claude Code？它是目前唯一一个将 AI Agent 能力完整暴露为 CLI 接口的工具。不依赖 IDE、不需要 GUI、不锁定编辑器——你的 shell 就是你的 IDE。这让它在 CI/CD 集成、远程服务器操作、批量自动化场景中拥有其他 AI 代理无法企及的优势。
核心理念——CLI 即 Agent

其他 AI 编程工具（Cursor、Copilot、Windsurf）本质是编辑器扩展，必须打开 GUI 才能使用。Claude Code 反其道而行——它是一个独立的命令行程序，可以在任何有终端的地方运行。

▸ 独立于编辑器
不绑定任何 IDE
不需要 VS Code、不需要 JetBrains。SSH 到远程服务器，打开终端就能用。这意味着在任何 Linux 服务器、Docker 容器、CI Runner 上都能直接调用 Agent 能力。
▸ 管道可组合
Unix 管道天然集成
Claude Code 支持 stdin/stdout，可与任何命令行工具组合。cat error.log | claude "分析这个错误"、git diff | claude "review 这段代码"——Unix 管道即调用接口。
▸ 脚本化调用
Shell 脚本中直接嵌入 Agent
在 bash 脚本中调用 claude -p "..." 就像调用任何 CLI 工具一样自然。批量处理、定时任务、CI pipeline 中都能无缝嵌入 AI 能力，不需要任何 SDK 封装。
▸ 多实例并行
开多个终端就是多 Agent
每个终端窗口即一个独立 Agent 会话。用 tmux 开多个 pane 并行处理多个 TASK，无需任何编排框架，终端复用器即多 Agent 调度器。

Hook 系统——Claude Code 的自动化引擎

Hook 是 Claude Code 的核心能力之一，允许在 Agent 特定动作前后自动执行 shell 命令。区别于 Git Hook，Agent Hook 拦截 AI 的行为并注入工程规范，其他 AI 编程工具不具备这种级别的行为控制能力。

Hook vs Git Hook：Git Hook 在 git 操作时触发；Claude Code Hook 在 Agent 操作时触发。例如，Agent 准备编辑文件时触发 PreToolUse，Agent 执行完 bash 命令后触发 PostToolUse。这使工程规范可在 Agent 执行过程中实时介入，而非事后检查。
HOOK 事件类型
PreToolUse — Agent 调用工具之前触发，可阻断不合规操作 PostToolUse — 工具执行完毕后触发，可注入补充信息 Notification — Agent 空闲等待时触发，适合发送通知 Stop — Agent 完成回合时触发，可做收尾动作 SubagentStop — 子 Agent 完成时触发
HOOK 匹配器（MATCHER）
"tool_name": "Edit" — 仅在编辑文件时触发 "tool_name": "Bash" — 仅在执行命令时触发 "tool_name": "Write" — 仅在创建文件时触发 不指定 matcher 则匹配所有工具调用
Hook 实战配置——settings.json
禁止 Agent 编辑受保护文件 安全防线 
// ~/.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": { "tool_name": "Edit" },
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -q '\"file_path\".*\\.env\\|credentials\\|secret'; then echo 'BLOCKED: 禁止编辑敏感文件' >&2; exit 2; fi"
          }
        ]
      }
    ]
  }
}
编辑后自动运行 lint 质量保障 
// 每次 Agent 编辑文件后自动格式化 + lint 检查
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": { "tool_name": "Edit" },
        "hooks": [
          {
            "type": "command",
            "command": "FILE=$(echo \"$CLAUDE_TOOL_INPUT\" | jq -r '.file_path') && npx prettier --write \"$FILE\" && npx eslint --fix \"$FILE\" 2>&1 | tail -5"
          }
        ]
      }
    ]
  }
}
Agent 完成后自动通知 团队协作 
// Agent 执行完毕后发送通知
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Agent 任务完成\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}
禁止安装未授权包 供应链安全 
// 拦截未经批准的 npm install
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": { "tool_name": "Bash" },
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -qE 'npm install|yarn add|pnpm add'; then echo 'BLOCKED: 需人工确认依赖安装' >&2; exit 2; fi"
          }
        ]
      }
    ]
  }
}
CLAUDE.md——项目级持久化指令

CLAUDE.md 文件是 Claude Code 的另一个独有能力：放在项目根目录的指令文件会在每次会话启动时自动加载。这相当于给 Agent 写了一份永久的「项目规章」，不需要每次都在提示词中重复。

CLAUDE.md 层级体系 自动加载 
# Claude Code 会按优先级自动加载以下文件：

~/.claude/CLAUDE.md # 全局指令（所有项目生效）
项目根/CLAUDE.md # 项目级指令（团队共享，提交到 git）
项目根/.claude/CLAUDE.md # 项目级私有指令（.gitignore）
当前目录/CLAUDE.md # 目录级指令（模块级别定制）

# 优先级：目录级 > 项目私有 > 项目级 > 全局
# 高优先级指令会覆盖低优先级的冲突配置

最佳实践——高效搭配 Claude Code 特性
▸ 实践 1
Hook + CLAUDE.md 双保险
CLAUDE.md 定义「应该怎么做」（编码规范、架构约束），Hook 定义「不许怎么做」（拦截违规操作）。指令是柔性引导，Hook 是硬性门控。两者搭配才能确保 Agent 产出始终合规。
▸ 实践 2
非交互模式 + CI 集成
在 GitLab CI / GitHub Actions 中用 claude -p "..." --output-format json 做自动化代码审查、变更分析、文档生成。Agent 的输出直接作为 CI 产物存档，或者用 jq 解析后驱动后续流程。
▸ 实践 3
tmux + 多 TASK 并行
用 tmux 开 4 个 pane，每个 pane 用 claude -p 执行一个独立的 TASK。每个会话有独立上下文，互不干扰。这比任何「多 Agent 框架」都更轻量、更可控——因为你能实时看到每个 Agent 在干什么。
▸ 实践 4
Skill 封装团队工作流
把团队反复执行的复杂流程封装为 Skill（如 /hx-go）。Skill 本质是一段预定义的提示词 + 执行逻辑，团队成员只需一条命令就能触发完整的标准化流程，不需要记住每个步骤。
▸ 实践 5
MCP Server 打通外部系统
通过 MCP（Model Context Protocol）Server 让 Claude Code 直接对接 DevOps 平台、数据库、消息系统。Agent 可以直接查询 Jira 任务、读写 GitLab API、查 Grafana 指标——不需要人工中转信息。
▸ 实践 6
会话恢复保上下文
复杂任务执行到一半需要暂停？用 claude --continue 恢复上次会话，所有上下文完整保留。不需要重新描述背景、不需要重新读取文件——Agent 接着上次停下的地方继续。
