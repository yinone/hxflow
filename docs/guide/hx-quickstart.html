框架是什么、怎么跑、怎么扩展

主路径步骤

一句话定位

HXFlow 不是业务框架，而是一个把“需求整理、任务拆解、执行、质检、交付”结构化的工程工作流框架。

一句话区分

hx 只负责安装链；hx-* / /hx-* 才是工程师每天使用的工作流命令。

一句话落地

机器级先装包并补齐入口，项目级再跑 hx-init，需求级优先走 hx-go 或主路径链路。

Memory

先记三句话

如果你只想在 10 分钟内建立正确心智，先把这三层概念记住，初学者就不会把 HXFlow 当成“另一个脚手架”或者“另一个代码框架”。

1 / 产品边界

它编排的是交付流程，不是业务运行时

仓库里没有 React、Vue、Spring 这类业务运行时框架代码，核心是 bin/hx.js、src/commands/、src/pipelines/ 和安装链工具。

2 / 真正入口

真正执行逻辑是 Markdown skill 契约

bin/hx.js 只处理 setup / migrate / version。像 hx-init、hx-doc、hx-go 以及维护类命令，本质上都是被 Agent 读取的 skill 契约文件。

3 / 扩展方式

扩展靠覆盖，不靠魔改内核

项目层 .hx/、用户层 ~/.hx/、系统层 src/ 形成统一查找顺序。团队一般在项目层或用户层改规则，而不是直接改内置 base。

i
阅读建议：先把“这不是代码框架，而是工程工作流框架”讲明白，再去看命令。否则新人会自然误以为 hx-run 是一个 shell 里的构建命令。
Repository

仓库地图：第一次进仓库该看哪里

下面这张地图直接对应当前仓库的真实目录。建议按“入口 → 规则 → 扩展 → 文档”的顺序阅读，不要从零散文件开始。

# 当前仓库核心结构
bin/hx.js                     # CLI 入口，只负责内置维护命令
src/scripts/hx-setup.js       # 安装与修复全局入口
src/scripts/lib/              # 安装链和命令元数据工具
src/commands/*.md             # hx-init / hx-doc / hx-go 等 workflow skill 契约
src/pipelines/default.yaml    # 默认主路径：doc -> plan -> run -> check -> mr
docs/guide/*.html             # 面向团队的静态文档
bin/hx.js

判断这是“维护 CLI”还是“业务工作流”

这个文件会明确告诉你：内置 CLI 命令只有 setup、migrate 和 version。如果你输入 hx-init，CLI 会提示这是 agent contract，而不是直接执行的 Node 子命令。

src/commands/

真正决定工作流行为的地方

每个 hx-* 命令都是一份 Markdown 约束文件，里面写“事实来源、执行步骤、边界和输出要求”。这是 HXFlow 最关键的设计点。

src/scripts/

系统层仅保留安装链

当前 Node 脚本只承担安装职责：用户手动运行 hx setup 完成初始化或修复。项目规则生成和维护动作交给 Agent 命令完成。

src/pipelines/default.yaml

一键执行为什么能跑起来

hx-go 会去读 pipeline，按步骤依次解析对应命令文件。所以一键模式本质上是“读取 YAML 后按顺序调度一组 Markdown 命令”。

✓
新人读仓库顺序：README.md → bin/hx.js → src/commands/hx-init.md → src/pipelines/default.yaml → src/scripts/hx-setup.js。
Mental Model

命令心智：先分清两套命令

最容易混淆的是“终端里的 hx 命令”和“Agent 里的 hx-* 命令”。这一页单独拆开讲，就是为了避免后续所有命令都被误用。

hx CLI

  • 运行环境：终端里的 Node CLI。
  • 当前仓库内置子命令：setup、version。
  • 职责：安装框架、修复 Agent 适配层。
  • 典型场景：新机器装一次、补装或修复入口。
npm install -g @hxflow/cli
hx setup

hx-* / /hx-* Agent 命令

  • 运行环境：Claude Code 或 Codex 这类 Agent 会话。
  • 职责：整理需求、产出计划、执行任务、做 QA、生成 MR 上下文，以及升级/卸载/诊断等维护动作。
  • 载体：src/commands/*.md 里的 workflow skill 契约。
  • 同一套 contract，Claude 用 /hx-*，Codex 用 $hx-*。
Claude: /hx-go
Codex: $hx-go
i
一句话区分：hx 只管安装链，hx-* 才是日常使用的工作流和维护命令。
Layers

三层扩展：为什么团队可以不改内核

HXFlow 的核心设计不是“所有团队都改同一个模板”，而是先统一全局运行规则，再通过三层同构目录做扩展。commands / pipelines 走覆盖解析，hooks 走中间件链执行。

项目层

<project>/.hx/。优先级最高，适合放项目专属命令、路径模板、Hook 和 pipeline。对业务团队来说，最常改的就是这里。

用户层

~/.hx/。跨项目共享，适合放个人或团队通用 Hook、自定义命令和 pipeline。装包后的自动 setup 会先把这层基础结构准备好。

系统层

<frameworkRoot>/src/。也就是当前仓库内置的默认实现：workflow skill 契约、默认 pipeline 和安装链脚本都在这里。

Command Lookup

命令优先级

例如 hx-go 会先找 .hx/commands/hx-go.md，再找 ~/.hx/commands/hx-go.md，最后才退回系统层。

Rules Lookup

项目规则

.hx/rules/*.md 由 /hx-init 生成骨架，团队在手动区写入规则后永久保留。/hx-rules update 可更新自动区，不影响手动区内容。

Pipeline Override

流程改造

默认主路径来自 src/pipelines/default.yaml，但项目层或用户层可以用同名文件覆盖，做到“保留框架骨架，但换执行顺序”。

✓
团队治理建议：不要直接改系统层 workflow skill 契约。团队规则应写入项目层 .hx/rules/ 的手动区，确保框架升级时不会被覆盖。
Usage

初学者怎么用：按时间尺度来记

不要直接背命令表。最容易记住的方式是按“机器级一次、项目级一次、需求级多次”来理解。

A
机器级：先装框架
全局安装 npm 包后，先手动运行一次 hx setup。这一步会写入 ~/.hx/ 目录结构、~/.hx/settings.yaml，并把同一套 workflow skill 安装到 ~/.claude/skills/ 与 ~/.agents/skills/。
B
项目级：初始化当前仓库
在 Agent 会话里运行 hx-init。它会分析项目结构、生成项目规则、写入 .hx/config.yaml、项目级目录骨架和默认 pipeline，并在 CLAUDE.md / AGENTS.md 中注入 Harness 标记块。
C
需求级：优先走主路径
最推荐的主路径是 hx-doc → hx-plan → hx-run → hx-check → hx-mr。不想手动分步时，直接用 hx-go 读取 pipeline 一次性调度这条链路。
最短上手路径

第一次体验建议这样讲

# 1. 机器只做一次
npm install -g @hxflow/cli --registry https://npm.cdfsunrise.com/
hx setup

# 2. 新项目做一次
Claude: /hx-init
Codex:  $hx-init

# 3. 进入需求交付
Claude: /hx-go
Codex:  $hx-go
手动模式

需要可控性时分步执行

Claude:
/hx-doc "用户登录页支持手机号登录"
/hx-plan
/hx-run
/hx-check
/hx-mr

Codex:
$hx-doc "用户登录页支持手机号登录"
$hx-plan
$hx-run
$hx-check
$hx-mr

主路径里通常不需要你手写 feature，因为 hx-doc 会自动生成，后续步骤会自动续接。只有冲突、恢复或多需求并行时才显式指定。

i
本仓库里的默认主路径事实：src/pipelines/default.yaml 当前定义的步骤就是 doc -> plan -> run -> check -> mr，其中 doc 和 plan 默认带 checkpoint，执行完后会等待用户确认。
Extension

团队怎么扩展：不要从源码乱改开始

建议把扩展点分成四类。这样新人既知道框架能改什么，也知道什么不该直接碰。

.hx/config.yaml

改路径和硬变量配置

这是项目级主入口。最先讲它，因为路径模板和质量门配置都在这里汇总。

.hx/rules/

改团队规则

当团队需要自己的架构约定、黄金原则时，编辑 .hx/rules/golden-rules.md 及其他规则文件。/hx-init 负责生成初始骨架，手动区内容永久保留。

.hx/commands/*.md

改命令行为

如果某个 skill 需要项目级改造，直接用同名 skill 文件覆盖。例如项目层写 .hx/commands/hx-run.md，优先级会高于系统层。

.hx/pipelines/default.yaml

改流程顺序

当团队不想走默认主路径，或者需要插入内部阶段时，优先改 pipeline，而不是去改 hx-go 的核心实现。

✓
记忆口径可以这样说：”想改规则，去 .hx/rules/；想改流程，去 pipeline；想改单个命令提示，去 commands；想改项目默认入口，去 config。” 这比逐个讲文件更容易记。
维护者验证

改完框架怎么自测

npm run hx:test
npm run hx:test:unit
npm run hx:test:integration

这些命令来自当前仓库 package.json。如果你的读者包含框架维护者，这三个脚本需要明确提到。

规则共享策略

共享策略

团队规则直接落在项目层的 .hx/rules/，由 /hx-init 和 /hx-rules update 维护。这样系统层升级时，团队手动写入的规则不会被覆盖。

Workshop

上手演练：一小时跑通一遍

如果你要带新人快速入门，建议把演练分成三个环节。每个环节都基于当前仓库的真实能力，不需要额外造例子。

20 分钟

环节一：理解框架结构

  • 打开 README.md，先建立整体概念。
  • 再看 bin/hx.js，分清维护 CLI 和 Agent 命令。
  • 最后浏览 src/commands/hx-init.md 和 src/pipelines/default.yaml。
20 分钟

环节二：跑一次主路径

  • 确认装包后的全局入口已经可用；若异常，再手动执行 hx setup 修复。
  • 项目级执行 /hx-init 或 $hx-init。
  • 需求级执行 /hx-go，或至少手动跑一遍 /hx-doc → /hx-plan。
20 分钟

环节三：做一次最小扩展

  • 在项目层编辑 .hx/rules/golden-rules.md，在手动区写入一条团队自己的规则。
  • 在 .hx/config.yaml 中添加一条 gates 配置。
  • 让大家理解”团队定制不是改源码，而是编辑项目规则文件”。
这套演练对新人的价值，不在于背会全部命令，而在于建立正确心智：哪里是系统层、哪里是团队层、哪里是项目层，什么改动应该落在哪一层。
Mistakes

常见误区：一开始就要纠正的错误理解

下面这些误区，都能直接从当前仓库代码里找到反证。提前讲清，能少掉大量后续答疑成本。

误区 1：hx-init 是普通 shell 子命令

不是。当前 bin/hx.js 会明确提示它是 agent command contract。日常工作流命令由 Agent 读取 Markdown 契约后执行。

误区 2：框架升级会覆盖团队规则

不会。团队规则存放在项目层的 .hx/rules/，由 AI 命令按当前项目事实更新；手动写入的团队规则不会因为系统层重装或升级而被覆盖。

误区 3：每次都要手动写 feature

不是主路径常态。hx-doc 会生成稳定的内部标识，后续命令通常自动续接。只有恢复执行、冲突消歧或多需求并行时才需要显式输入。

误区 4：hx-go 是写死的流程

不是。它读的是 pipeline YAML，所以团队完全可以覆盖 .hx/pipelines/default.yaml 来改变流程顺序或插入检查点。
